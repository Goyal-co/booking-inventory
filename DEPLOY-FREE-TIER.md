# Deploy on Free Tier (Neon + Render)

This guide deploys the full stack **for $0/month** using:

| Service | Free tier | Role |
|---------|-----------|------|
| [Neon](https://neon.tech) | 512 MB Postgres | Database |
| [Render](https://render.com) | 3 web services | Admin, Sales, WebSocket |

You get default URLs like `https://booking-inventory-sales.onrender.com` ‚Äî add a custom domain later in Render/Neon settings.

**Free tier limits to expect:**

- Render services **sleep after ~15 min** of no traffic (30‚Äì60s cold start)
- Neon free DB has storage/compute caps (fine for pilots)
- No persistent disk ‚Äî uploaded floor plans reset on redeploy (use S3 later if needed)

## Step 1 ‚Äî Push code to GitHub

If the repo is not on GitHub yet:

```bash
git init
git add .
git commit -m "Prepare for Render deployment"
git branch -M main
git remote add origin https://github.com/YOUR_USER/booking-inventory.git
git push -u origin main
```

Do **not** commit `.env` files (already in `.gitignore`).

---

## Step 2 ‚Äî Create Neon database (free)

1. Sign up at [neon.tech](https://neon.tech)
2. **New Project** ‚Üí name it `booking-inventory`
3. Open **Connection details** ‚Üí copy the **pooled** connection string
4. Append SSL if missing:

```
postgresql://USER:PASSWORD@HOST/neondb?sslmode=require
```

Save this as `DATABASE_URL` ‚Äî you will use it locally and on Render.

---

## Step 3 ‚Äî Prepare database & Super Admin (from your PC)

In PowerShell (Windows) from the project root:

```powershell
# Paste your Neon connection string
$env:DATABASE_URL = "postgresql://....?sslmode=require"

pnpm install
pnpm db:generate
pnpm db:push

# Production bootstrap ‚Äî choose YOUR admin email/password
$env:NODE_ENV = "production"
$env:ORGANIZATION_NAME = "Your Company"
$env:ORGANIZATION_SLUG = "your-company"
$env:SUPER_ADMIN_EMAIL = "you@email.com"
$env:SUPER_ADMIN_NAME = "Super Admin"
$env:SUPER_ADMIN_PASSWORD = "YourStrongPassword12!"

# Generate secrets (save these for Render)
# Run in Git Bash or use an online generator ‚Äî need 32+ char random strings:
# NEXTAUTH_SECRET_ADMIN, NEXTAUTH_SECRET_SALES, WS_INTERNAL_SECRET

pnpm db:bootstrap
```

You should see `Bootstrap complete` with your admin email.

**Do not** run `SEED_DEMO=true pnpm db:seed` in production.

---

## Step 4 ‚Äî Generate secrets

Run three times (or use [1Password generator / openssl](https://openssl.org)):

```bash
openssl rand -base64 32
```

Save as:

| Secret | Used on |
|--------|---------|
| `NEXTAUTH_SECRET` (admin) | Admin service only |
| `NEXTAUTH_SECRET` (sales) | Sales service only ‚Äî **different value** |
| `WS_INTERNAL_SECRET` | WS + Admin + Sales |

---

## Step 5 ‚Äî Deploy on Render (Blueprint)

1. Sign up at [render.com](https://render.com) ‚Üí connect GitHub
2. **New** ‚Üí **Blueprint** ‚Üí select this repository
3. Render reads `render.yaml` and creates 3 services
4. When prompted, set **`DATABASE_URL`** (same Neon string) on **all three** services
5. Click **Apply**

First deploy will start. WS may succeed first; admin/sales need URL env vars (next step).

---

## Step 6 ‚Äî Set public URLs (after first deploy)

Open each service in Render ‚Üí **Environment**. Note the public URLs:

| Service | Example URL |
|---------|-------------|
| WS | `https://booking-inventory-ws.onrender.com` |
| Admin | `https://booking-inventory-admin.onrender.com` |
| Sales | `https://booking-inventory-sales.onrender.com` |

Set these environment variables:

### `booking-inventory-ws`

| Key | Value |
|-----|-------|
| `CORS_ORIGINS` | `https://booking-inventory-sales.onrender.com,https://booking-inventory-admin.onrender.com` |

(`WS_INTERNAL_SECRET` is auto-generated by Blueprint; copy it if you deploy manually.)

### `booking-inventory-admin`

| Key | Value |
|-----|-------|
| `NEXTAUTH_URL` | `https://booking-inventory-admin.onrender.com` |
| `WS_EMIT_URL` | `https://booking-inventory-ws.onrender.com` |
| `WS_INTERNAL_SECRET` | Same as WS service |

If Blueprint linked `WS_INTERNAL_SECRET` from WS, you only set `NEXTAUTH_URL` and `WS_EMIT_URL`.

Use a **unique** `NEXTAUTH_SECRET` if Render generated one per service ‚Äî that is fine.

### `booking-inventory-sales`

| Key | Value |
|-----|-------|
| `NEXTAUTH_URL` | `https://booking-inventory-sales.onrender.com` |
| `NEXT_PUBLIC_WS_URL` | `https://booking-inventory-ws.onrender.com` |
| `WS_EMIT_URL` | `https://booking-inventory-ws.onrender.com` |
| `WS_INTERNAL_SECRET` | Same as WS service |

**Important:** `NEXT_PUBLIC_WS_URL` is baked in at **build** time. After setting it, trigger **Manual Deploy ‚Üí Clear build cache & deploy** on the sales service.

---

## Step 7 ‚Äî Redeploy

1. **Manual Deploy** on all three services (or push a git commit)
2. Wait for builds to finish (5‚Äì10 min first time)

---

## Step 8 ‚Äî Verify

1. **Admin:** open `https://booking-inventory-admin.onrender.com/login`  
   Log in with `SUPER_ADMIN_EMAIL` / `SUPER_ADMIN_PASSWORD`
2. Create a project, add inventory
3. **Sales:** open sales URL ‚Üí log in as a sales user you create in Admin ‚Üí Users
4. Block a unit ‚Äî grid should update live (WebSocket)

If live updates fail, check `NEXT_PUBLIC_WS_URL`, `CORS_ORIGINS`, and `WS_INTERNAL_SECRET` match across services.

## Manual deploy (without Blueprint)

Create three **Web Services** on Render:

| Setting | WS | Admin | Sales |
|---------|----|-------|-------|
| Root Directory | `.` | `.` | `.` |
| Build Command | `corepack enable && pnpm install --frozen-lockfile && pnpm db:generate` | `corepack enable && pnpm install --frozen-lockfile && pnpm db:generate && pnpm --filter admin build` | `corepack enable && pnpm install --frozen-lockfile && pnpm db:generate && pnpm --filter sales build` |
| Start Command | `pnpm --filter ws-server start` | `pnpm --filter admin start` | `pnpm --filter sales start` |
| Health Check | `/health` | ‚Äî | ‚Äî |

Use the same env vars as Step 6.

## Adding a custom domain later

1. Render ‚Üí each service ‚Üí **Settings** ‚Üí **Custom Domains**
2. Add e.g. `admin.yourdomain.com`, `app.yourdomain.com`, `ws.yourdomain.com`
3. Update DNS CNAME records as Render instructs
4. Update `NEXTAUTH_URL`, `NEXT_PUBLIC_WS_URL`, `WS_EMIT_URL`, `CORS_ORIGINS` to HTTPS domain URLs
5. Redeploy sales (for `NEXT_PUBLIC_WS_URL` rebuild)

## Troubleshooting

| Problem | Fix |
|---------|-----|
| Build fails on Prisma | Ensure `DATABASE_URL` is set on Render (needed for install; generate does not connect) |
| Admin login fails | Confirm bootstrap ran; check Neon tables in Neon SQL editor |
| `Invalid NEXTAUTH_SECRET` | Secret must be 32+ chars; set on each app |
| Live grid never updates | Redeploy sales after setting `NEXT_PUBLIC_WS_URL`; check browser devtools ‚Üí Network ‚Üí WS |
| Cold start slow | Normal on free tier ‚Äî first request wakes the service |
| Uploads disappear | Render free tier has ephemeral disk ‚Äî use external storage for production PDFs |

---

## Security reminders

- Never commit secrets or `.env`
- Remove `SUPER_ADMIN_PASSWORD` from your local shell history after bootstrap
- Use different `NEXTAUTH_SECRET` for admin vs sales
- Do not enable `SEED_DEMO` on Render

See also [DEPLOYMENT.md](./DEPLOYMENT.md) for the full security checklist.
